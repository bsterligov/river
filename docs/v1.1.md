# How it works V1.1 тАФ Workflow refinements

V1.1 is not a new agentic loop тАФ it is a set of refinements to the V1 workflow that reduce CI noise, tighten quality gates, and make the mise task structure easier to navigate.

## What changed

### 1. SonarQube moved out of CI

SonarQube now runs as a step inside `spec-dev`, not as a CI job on every PR.

**Before:** every impl PR triggered a full CI pipeline that included coverage generation, artifact upload, and a SonarQube scan. Draft PRs were skipped, but any `ready_for_review` push ran the full scan тАФ even for trivial changes.

**After:** SonarQube is an explicit step in `spec-dev` that the agent runs after the code is stable, before marking the PR ready. The agent sees the quality gate result and fixes issues in the same session, without waiting for a CI queue. CI (lint + tests) still fires when the PR moves from draft to ready тАФ serving as an independent reproducible check, not a first-pass gate.

`spec-dev` Step 4:
```bash
mise run ci:sonar-scan   # generates coverage + submits PR scan
mise run ci:sonar-check  # polls gate, prints any failing conditions
```

### 2. One impl PR at a time

`spec-dev` now blocks if another `impl/*` PR is already open before creating a new one.

```bash
mise run agent:check-impl-pr
```

This enforces the single-PR-in-flight convention that was previously just documented. Parallel plan-dev subagents are unaffected тАФ they each work in isolated git worktrees and create PRs on separate branches. The guard only fires in single-task `spec-dev` runs where opening a second PR would be accidental.

### 3. Mise task structure refactored

All tasks moved from nested namespace directories (`cargo/`, `flutter/`, `sonar/`, `spec/`, `release/`, `index/`) to a flat two-bucket structure:

| Bucket | Purpose | Task naming |
|--------|---------|-------------|
| `ci/` | Bash tasks that run in CI or locally to validate | `ci:cargo-clippy`, `ci:flutter-test`, `ci:sonar-scan`, тАж |
| `agent/` | Agentic and developer workflow tasks | `agent:spec-draft`, `agent:flutter-client`, `agent:index-reindex`, тАж |

**Full task list:**

`ci/`: `cargo-clippy` ┬╖ `cargo-coverage` ┬╖ `cargo-fmt-check` ┬╖ `cargo-test` ┬╖ `flutter-analyze` ┬╖ `flutter-ci-setup` ┬╖ `flutter-coverage` ┬╖ `flutter-test` ┬╖ `release-version` ┬╖ `release-publish` ┬╖ `sonar-scan` ┬╖ `sonar-check`

`agent/`: `spec-draft` ┬╖ `spec-impl` ┬╖ `spec-publish` ┬╖ `flutter-client` ┬╖ `flutter-build` ┬╖ `flutter-run` ┬╖ `flutter-setup` ┬╖ `index-reindex` ┬╖ `check-duplication` ┬╖ `check-impl-pr` ┬╖ `sonar-issues` ┬╖ `calculate-usage`

The distinction is intentional: `ci/` tasks are short, stateless, and suitable for CI runners; `agent/` tasks are interactive or side-effectful (open PRs, regen clients, reindex the workspace).
